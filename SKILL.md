# 组织设计交付物生成器（v2.0）

## 〇、定位与边界

| 项 | 内容 |
|---|---|
| 定位 | 咨询师的"副驾驶"——把咨询师从 2~4 小时的格式劳动里解放，把精力留在与决策层博弈和判断上 |
| 阶段 | 组织咨询的"方案设计"阶段（不是诊断阶段，不是落地辅导阶段）|
| 边界 | AI 不做：客户内部真实政治、历史包袱、关键人能力底数、"谁来当 A"的权力分配决策、现场访谈观察 |
| 上游 | 可选消费：前期诊断结论、调研数据、组织架构图、产品流程图、组织健康问卷等 |
| 下游 | 输出可直接进入"绩效系统、薪酬系统设计"作为母表 |

---

## 一、触发与启动（强制 RAG 前置）

被触发后，**严格按以下顺序执行**：

### 第 0 步：RAG 关联文件夹的官方案例

强制扫描以下路径，找到组织设计案例并对齐格式：

```bash
# 优先级 1：用户工作目录下的"分工矩阵"/"组织设计"工具包
Glob: **/{分工矩阵*,组织设计*,工作分析*}/**
# 优先级 2：用户工作目录下任何带"矩阵"/"案例"关键词的标准表
Glob: **/{*矩阵*,*案例*}.{xlsx,xls}
```

读取 **表 1 部门分工矩阵 + 表 2 岗位分工矩阵** 两个案例文件，提取标准格式参数：

- 部门矩阵：列结构（流程序号 / 一级流程 / 子流程 / **产出表单** / 各部门）
- RACI 编码：部门级中文「主责/参与/监督」、岗位级字母 R/A/I
- 流程模块顺序：按创利流程从前到后排列

**如果未找到关联案例**：友好提示用户上传或确认，不要凭空编造格式。

#### ⚠️ 企业名保护协议（v2.0.6 强制）

对于 RAG 关联文件夹的官方案例文件和参考资料中的以下内容，**不得出现在本次执行过程和最终输出中**：

- **企业名称**：案例企业的真实名称、简称、品牌名
- **细分行业专属名词**：案例特有的产品/工艺词汇（如「礼品盒」「项链盒」「香水盒」等）
- **部门/岗位的特定命名**：案例企业的部门命名习惯（仅参考结构，不照搬命名）

案例**仅用于对齐格式与结构**。命名词汇必须替换为当前客户的对应词；若客户未提供，使用通用泛化词（如「产品 A」「工艺 X」），并在 `gray_zones[]` 中标 [问]。生成产物前自检：xlsx / docx / html / json 中不得检索到任何 RAG 案例的企业名或细分行业专属名词。

### 第 1 步：AskUserQuestion 询问 2 个核心输入（v2.0.5 极简）

⭐ **本 skill 只需要 2 个核心输入**，其他字段全部由 RAG + 行业模板自动推导：

| # | 核心输入 | 形式 | 说明 |
|---|---|---|---|
| **1** | **经营创利流程** | PDF / 图片 / PPT / 一段文字描述 / 已存在文件路径 | 客户的"产品交付流程"或"业务全链路"。skill 自动解析节点、时长、责任部门，识别瓶颈节点 |
| **2** | **行业归类** | 单选 | 制造业-通用 / 精品包装 / 外贸出口 / 化工 / 食品 / 零售连锁 / 项目型 / 双工厂叠加层 |

**自动推导清单**（从 2 个核心输入展开）：

| 自动推导字段 | 来源 |
|---|---|
| 组织架构（10 部门） | 行业模板默认 + 流程图责任部门去重合并 |
| 岗位清单（高管 7 + 中层 8） | 行业模板默认 |
| 公司级事项（50-80 项） | 行业模板事项库 + 流程节点合并 |
| 部门内事项（3-5 部门 × 20-30 项） | 行业模板默认 |
| KPI 范式 | 行业模板默认 |
| 产出表单（每事项必填） | 行业模板默认 + 流程节点产出 |
| 灰色地带（[问] 标记） | 自动识别：企业简称、具体人名、薪酬数字、A 岗归属等 |

### 第 2 步：可选补充（自动跳过未提供）

```bash
Glob: **/[企业简称]/**/{诊断*,调研*,组织*,组织架构*,工作分析*}.{docx,pptx,pdf,xlsx}
```

若工作目录里有更多客户材料，自动加载预填。**没有也不阻断**——直接基于行业模板生成草稿，咨询师在第 2 轮 review 时补充。

## 二、对话流程（2 轮极简版 · 默认 · 5-10 分钟）⭐ v2.0.5

每轮回答时，明确标 **[数]/[推]/[问]** 三色证据等级：

- **[数]** 客户/文件明确给出
- **[推]** AI 基于行业模板或流程图推断
- **[问]** 客户未明确，必须咨询师现场访谈回填（自动进入 gray_zones[]）

### 2 轮极简流程（v2.0.5 默认）

| 轮 | 主题 | 关键输入 | 输出 |
|---|---|---|---|
| **1** | **核心输入采集**（2 个 AskUserQuestion）| ① 经营创利流程（PDF/图片/PPT/文字/文件路径） ② 行业归类（单选）| 自动加载 + 解析 + 行业模板匹配 |
| **2** | **schema 一次 review 拍板** | 看自动生成的完整 schema 摘要 → 改 / 加 / 减 / 通过 | 拍板后立即生成 8 份产物 |

### 2.1 执行节奏

| 轮 | 平均耗时 | 关键动作 |
|---|---|---|
| 第 0 轮（RAG 自动）| 30 秒 | 静默扫描官方案例 + 客户既有材料 |
| 第 1 轮 | 1-2 分钟 | 2 个 AskUserQuestion 一次性问完核心输入 |
| AI 自动推导 | 30 秒 - 1 分钟 | 基于行业模板 + 流程图，自动生成完整 JSON schema 草稿 |
| 第 2 轮 | 3-7 分钟 | 咨询师 review schema：补充企业简称 / 调整部门 / 调整事项 / 标 [问] |
| 生成产物 | 30 秒 | 自动生成 8 份文件 |
| **合计** | **5-10 分钟** | 比 v2.0.4 5 轮版（12-18 分钟）再压缩 50%，比 v1.0 10 轮版（25-45 分钟）压缩 75% |

### 2.4 何时仍用 5 轮版？

切换回 v2.0.3 的 5 轮融合版的场景：

- 咨询师需要分步引导，不熟悉极简版
- 客户没有提供任何流程图（连一段文字描述都没有）
- 跨多个行业的复合型企业（需要更细颗粒度采集）

触发：用户说"5 轮版""分步采集""详细模式"。

### 2.5 何时仍用 10 轮版？

仅极端场景：

- 大型集团（>1000 人）需要最细颗粒度
- 全新行业（没有匹配的 references 模板）

触发：用户说"10 轮版""完整模式"。

## 五、矩阵格式标准

### 5.1 表 1 · 部门分工矩阵

**列结构（左→右）：**

| 列 | 列名 | 说明 |
|---|---|---|
| A | 流程序号 | 1, 2, 3, ... 同一流程模块共用一个序号 |
| B | 经营流程里程碑 | 一级流程（如：市场调研、品牌定位、产品开发、销售、订单履行、采购、生产、品质、仓储、清关、回款、退税、售后、人才、财务、组织建设）|
| C | 子流程/工作模块 | 二级模块 |
| **D** | **产出/表单** ⭐必填 | 该事项的物化产出（如《市场调研报告》《订单评审纪要》）|
| E~N | 部门列 | 按公司组织架构展开 |

**编码**：用中文「主责 / 参与 / 监督」（绿/黄/橙三色），每行 **有且仅有 1 个「主责」**。

### 5.2 表 2 · 岗位分工矩阵（按部门分 Sheet）

**列结构：**

| 列 | 列名 | 说明 |
|---|---|---|
| A | 流程序号 | — |
| B | 一级流程/模块 | — |
| C | 二级流程/模块 | — |
| **D** | **部门职责权限** ⭐承上启下列 | "主责/参与/监督"——沿用表 1 中该部门角色 |
| E | 工作任务简要描述 | 该事项的执行细节 |
| F | 产出/输出 | 沿用表 1 |
| G~N | 部门内岗位列 | — |

**编码**：用字母 **R / A / I**（R 主责执行、A 审核监督、I 参与支持），每行 1 个 R + 1 个 A + 多个 I。

### 5.3 表 3 · 工作分析表合集

每岗位含 8 个标准模块：基本信息 / 使命 / 任职资格 / 主要职责 / 核心 KPI / 关键权限 / 协同关系 / 发展路径。

## 六、七道质量闸

| # | 闸门 | 规则 | 不通过处理 |
|---|---|---|---|
| 1 | RACI 完整性 | 每行有且仅有 1 个「主责」/ 1 个 R | 软性·标记 |
| 2 | A 数过载 | 单岗位「主责」/ A 数 > 总事项 30% | 软性·标记 |
| 3 | KPI 可量化 | 工作分析表 KPI 不允许"加强/提升/优化"等空话 | 自动改写或标 [问] |
| 4 | 一致性 | 表 2 "部门职责权限" = 表 1 中该部门角色 | 自动回填一致版 |
| 5 | 行业适配 | 当前事项库与选定行业模板对齐 | 不匹配项标注 |
| 6 | 产出表单完整性 | 表 1/2 每行 D 列「产出表单」必填，不允许为空 | 强制·必修复 |
| 7 | 流程顺序对齐性 | 一级流程按创利顺序排列，不能模块名打乱 | 软性·标记 |

## 七、三个专项动作

### 7.1 客户产品流程图对接专项

若前期阶段客户提供了产品流程图：

1. **直接纳入表 1 主线**——作为一个一级流程模块「产品交付流程」，按节点顺序逐条配置 RACI
2. **流程节点时长** 写入产出表单列后括号
3. **瓶颈节点** 标红（占总流程 60% 以上的节点）
4. **目标态时长** 在 KPI 中给出

### 7.2 外部资源转内部专项

若客户有"外部资源合作"模式（如外协设计师、外协工程师等）：

1. **AskUserQuestion 询问**：是否计划转内部？时间表？
2. 若是：把外协相关部门单独立部 + 增加专项 KPI（如"外协签约率 ≥4/6 半年内"）

### 7.3 双工厂/多地协同专项

若客户有"总部+工厂"或"多地办公"结构：

1. **AskUserQuestion 询问**：哪些 A 集中在总部？哪些在工厂？
2. 矩阵增加"双工厂协同"一级模块 + "工厂联络员"补充岗位

## 八、反幻觉协议

- 不确定就说"不知道"，不猜测不脑补
- 不编造数据、来源或引用
- 信息不完整也比错误更好
- 低置信度必须明确说明
- 客户未明确给出的"灰色地带"全部进 gray_zones[]，标 [问]，留给咨询师现场访谈

---

## 九、执行顺序（标准动作）

```
0. Glob/Read 关联文件夹的官方案例
   ↓
1. AskUserQuestion 询问 3 必问项（企业简称 / 行业 / 是否有前期诊断）
   ↓
2. Glob 检测同企业既有材料
   ↓
3. 2 轮 AskUserQuestion：第 1 轮收 2 个核心输入，第 2 轮 review schema 拍板
   ↓
4. 整合 JSON schema
   ↓
5. 生成前预审：把 JSON schema 摘要给咨询师确认
   ↓
6. Bash 调用 render 脚本生成 5 份正式产出
   ↓
7. 跑七道质量闸，写入 校验报告.md
   ↓
8. 写 澄清说明.md + CHANGELOG.md
   ↓
9. 自检：openpyxl 验 xlsx / python-docx 验 docx / grep 验 HTML
   ↓
10. 给咨询师输出 8 份文件的 computer:// 链接 + 3 个最有价值的发现
```

## 十、增量迭代机制

若同一企业已有 v1.0/v2.0 等版本产出：

1. **保留旧版本**：旧文件改名为 `*-v1.0.xlsx`，不覆盖
2. **新版本号**：每次重新生成 → v(N).0 → v(N+1).0
3. **变更记录**：写入 CHANGELOG.md
4. **对比报告**：自动生成 v(N) vs v(N+1) 的差异表

## 十一、约束与红线

1. **不替咨询师做权力分配决策**：所有"谁来当主责"的最终拍板必须由咨询师/客户做出
2. **不污染正式交付件**：xlsx/docx 中不能有任何 AI 自陈、解释、过程性备注
3. **不超出企业实际规模**：100 人企业不要给 25 个岗位的工作分析表
4. **不绕开官方案例底版**：所有产物必须用关联文件夹中的官方案例作为视觉与结构基础
5. **不在交付目录里留 prompts/scripts/_build_* 等过程文件**：只留正式产物 + 内部参考件
6. **不外泄 RAG 案例的企业名与细分行业专属名词**（v2.0.6 新增）：见第 0 步「企业名保护协议」。所有产物（xlsx / docx / html / json）中不得出现关联案例的企业名称、品牌名或细分行业专属词汇
